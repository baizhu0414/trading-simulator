### 2. 命令执行过程与流量变化分析
本次测试执行过程分为两个阶段，流量呈现**“瞬间冲高但迅速遇阻”**的特征：

#### 阶段 1：用户极速孵化（0s - 0.01s）
- **目标**：100并发用户。
- **动作**：由于 `-r 100`（孵化速率100用户/秒），系统在**10毫秒内**（18:47:21.213 至 18:47:21.220）瞬间完成了全部 100 个虚拟用户的启动。
- **预期流量**：如果脚本使用 `between(0.05, 0.1)` 且服务器响应够快，此时应迅速达到 **500-2000 RPS** 的流量峰值。

#### 阶段 2：流量遇阻瓶颈期（0.01s - 10s）
- **实际表现**：流量并未如预期爆发，而是被严重压制。
- **原因**：服务器处理速度太慢，100个用户发出请求后，大部分都在**等待响应**（阻塞），无法发出下一个请求。
- **结果**：整个测试期间，实际流量曲线非常平缓，维持在 **50 RPS** 左右，未能形成冲击。

## 四、服务器状态推测与瓶颈分析
根据“吞吐量低、响应慢、但无报错”的现象，推测服务器端可能存在以下情况：

### 1. 最可能的瓶颈：数据库写入阻塞
- **现象**：订单接口（需要写库、撮合、更新状态）比撤单接口（逻辑简单）慢 7 倍以上。
- **推测**：你的数据库持久化逻辑可能是**同步写入**，或者数据库连接池太小。
- **建议排查**：
  1. 检查数据库 CPU/IO 使用率。
  2. 检查代码中 `persistOrderAndTrades` 方法是否为 `@Async` 异步执行。
  3. 如果是同步写入，每笔订单都在等数据库落盘，1秒的响应时间是合理的（但不可接受）。

### 2. 次可能的瓶颈：Java GC（垃圾回收）或锁竞争
- **现象**：响应时间波动极大（从 97ms 到 3363ms）。
- **推测**：
  1. 系统在运行过程中频繁创建 `Order` 对象，导致年轻代 GC 频繁，每次 GC 会“Stop The World”（停止所有工作线程）几十到几百毫秒。
  2. 撮合逻辑中的 `ConcurrentSkipListMap` 或 `synchronized` 锁竞争激烈，导致线程排队等待。

### 3. 服务器资源情况
- **CPU**：可能不高（因为都在等 IO 或锁）。
- **内存**：可能有波动（GC 导致）。
- **网络**：由于是本地/内网测试，带宽通常不是问题。

## 五、总结与建议
### 1. 测试总结
1.  **功能稳定性**：✅ 优秀。100并发下无失败请求。
2.  **性能表现**：❌ 不达标。
    *   目标：500-1000 RPS。
    *   实际：52 RPS。
    *   差距：约 10-20 倍。
3.  **核心痛点**：订单接口响应太慢（中位数 1000ms），导致并发用户都在“排队”，无法形成有效流量压力。

### 2. 下一步优化建议（按优先级）
1.  **立即优化：数据库异步批量写入**
    *   不要每笔订单都实时写库。
    *   使用 `@Async` + `BlockingQueue` 收集订单，每 100ms 或每 100 笔批量写入一次。
2.  **排查 GC 情况**
    *   启动 Java 应用时加上 GC 日志参数：`-Xlog:gc*:file=gc.log`。
    *   跑完压测后看 `gc.log`，如果有频繁的 Full GC，需要调整堆内存大小。
3.  **优化撮合锁粒度**
    *   检查 `OrderBook` 中的锁，是否可以按股票代码分段加锁，而不是全局锁。

### 3. 后续测试建议
在优化代码之前，不需要再提高并发数（加到 200 或 500 也没用，只会让排队更长）。先把订单接口的响应时间优化到 **200ms 以内**，再冲击 500 RPS 的目标。

## 前置通用规则（所有案例强制执行）
1.  **标的规范**：统一使用任务书指定标的 `600030(XSHG)`、`600000(XSHG)`、`000001(XSHE)`，避免标的分散导致对账复杂
2.  **股东号隔离**：买单专用股东号 `SH00000001~SH00000010`，卖单专用股东号 `SH00000011~SH00000020`，天然规避对敲风控，保证主流程压测流量有效
3.  **订单ID规范**：严格16位全局唯一，买单`1+15位序号`、卖单`1+15位序号`、撤单`1+15位序号`，符合任务书数据结构要求
4.  **价格规则**：主流程案例统一使用`10.00±0.2`固定价格区间，保证买卖价格匹配，最大化成交率，贴合“成交主导”的测试需求
5.  **数量规则**：主流程案例均为100整数倍委托，符合A股整手交易规则，避免无效拦截
6.  **核心对账恒等式（一票否决）**
    - 全市场维度：`累计买入成交总量 ≡ 累计卖出成交总量`
    - 单笔订单维度：`申报数量 = 累计成交数量 + 已撤数量 + 剩余未成交数量`
    - 终态约束：`CANCELED/FULL_FILLED/REJECTED/RISK_REJECT` 终态订单不可再发生状态变更、成交或撤单

## 第一大类：高并发主流程压测案例集（占比80%，核心测试项）
本大类为压测核心流量来源，重点测试系统吞吐量、数据库性能瓶颈、高并发下撮合数据一致性，Locust可直接按比例适配并发请求。

### 场景1.1 固定价格全量成交并发场景（占主流程流量50%）
**测试目标**
1.  验证高并发下买卖订单全量撮合的系统吞吐量、响应时延，定位数据库写入性能瓶颈
2.  验证全量成交场景下买卖累计成交量完全相等，数据一致性零偏差
3.  验证高并发下价格优先、时间优先规则的正确性

**前置条件**
1.  系统启动完成，订单簿清空，`t_exchange_order`、`t_exchange_trade` 表数据清空
2.  零股开关关闭：`matching.zero-share-enable=false`
3.  对敲风控开关开启：`risk.self-trade-enable=true`
4.  20个测试股东号无历史挂单

**并发输入设计**
- 买卖单数量比例1:1，保证全量成交
- 单批次并发量梯度递增：100→500→1000→5000→10000笔/秒
- 固定撮合价格10.00元，所有订单价格完全匹配
- 委托数量：随机100/200/500/1000股（均为100整数倍）
- 标的：随机选择3只合规股票
- 并发模式：买卖单同时并发提交，模拟真实交易双向报单

**单订单输入示例**
```

{
```

**强制对账校验点**
1.  全市场维度：`累计买单申报总量 = 累计卖单申报总量 = 累计买入成交总量 = 累计卖出成交总量`
2.  订单维度：所有订单状态均为`FULL_FILLED`，无剩余未成交、无撤单数量
3.  成交表维度：同一`execId`的买卖成交记录成对出现，`execQty`完全相等
4.  数据库约束：无重复`clOrderId`，无唯一约束冲突报错

**性能关注点**
- 接口平均响应时延、P99时延
- 每秒订单处理吞吐量（TPS）
- 数据库CPU、IO使用率，定位是否为性能瓶颈
- 无超时、无数据库连接池耗尽、无Redis不可用报错

### 场景1.4 下单-成交-撤单混合并发场景（占主流程流量10%）
**测试目标**
1.  验证高并发下下单、撮合、撤单同时进行时，系统稳定性和数据一致性
2.  验证撤单与撮合的并发竞争，已撮合订单不得被撤单，已撤单订单不得被撮合
3.  测试混合场景下数据库读写性能瓶颈

**前置条件**：同场景1.1，订单簿与业务表清空

**并发输入设计**
- 流量比例：下单70%、成交20%、撤单10%
- 下单：并发提交买卖单，价格10.00元，保证可撮合
- 撤单：从已提交未成交订单中，随机选择10%发起撤单
- 并发模式：下单、撮合、撤单同时进行，无先后顺序，模拟真实交易混合流量

**预期输出与校验规则**
1.  撤单成功的订单，状态为`CANCELED`，不得参与后续撮合，无成交记录
2.  已完全成交的订单，撤单请求返回`5004`（订单状态不可撤销）
3.  部分成交的订单，撤单成功后，剩余未成交数量不得参与撮合，累计成交数量不变
4.  对账校验：`累计申报数量 = 累计成交数量 + 累计撤单数量 + 剩余未成交数量` 恒成立

## 并发测试执行与结果校验指南
### 一、Locust压测脚本适配建议
1.  流量比例：严格按照主流程80%、一致性专项10%、边界场景7%、复杂流程3%分配
2.  订单ID生成：严格16位全局唯一，避免重复订单号导致无效压测
3.  股东号隔离：买卖单使用不同股东号，避免对敲风控拦截导致成交率不足
4.  结果收集：不仅记录接口响应，还要同步记录成交数量、订单状态，便于事后对账
5.  梯度压测：从100并发开始，逐步提升到500/1000/5000/10000，每梯度稳定运行5分钟，记录性能指标，定位数据库瓶颈

### 二、测试结果一票否决标准
1.  **数据一致性不达标**：买卖累计成交量不相等、单笔订单数量恒等式不成立、终态订单发生变更
2.  **业务规则失效**：价格优先/时间优先规则错乱、对敲风控漏拦截、整手规则漏放
3.  **稳定性不达标**：系统崩溃、OOM、数据库死锁、连接池耗尽、超时率>0.1%
4.  **幂等性失效**：重复订单号重复入库、重复成交、重复撤单

### 三、并发问题快速定位指南
1.  **对账不平**：优先检查部分成交订单的剩余数量、撤单数量，是否存在“既成交又撤单”脏数据、重复成交记录
2.  **撮合顺序错乱**：检查订单簿排序逻辑，是否并发下时间戳排序错误、锁机制导致顺序错乱
3.  **性能瓶颈**：通过监控定位数据库CPU/IO、Redis响应时延、接口P99时延，确认瓶颈点
4.  **幂等性失效**：检查`clOrderId`唯一约束是否生效、是否使用分布式锁保证幂等
